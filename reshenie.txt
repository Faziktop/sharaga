## Часть 1. Методологии User Story Mapping и Impact Mapping (кратко)

**User Story Mapping** — метод формирования бэклога продукта через визуализацию шагов пользователя («activities» → «tasks» → «stories»). Позволяет увидеть целостную картину взаимодействия с системой, выделить MVP и зоны для доработки.

**Impact Mapping** — стратегический метод планирования, связывающий бизнес-цель («Why») → действующих лиц («Who») → способы влияния («How») → функции («What»). Помогает увязать функционал с измеримыми результатами для бизнеса.

## Часть 2. Таблица функций и функциональная модель

### Таблица функций ИС (МИС стоматологии)

| Функция | Описание |
|---------|----------|
| Запись пациентов | Создание, перенос, отмена визитов; автоматическая рассылка напоминаний |
| Ведение электронной карты | Хранение анамнеза, диагнозов, снимков, планов лечения, согласий |
| Учёт неявок | Фиксация пропусков, автоблокировка записи при частых неявках, формирование штрафов/предупреждений |
| Управление расписанием врачей | Настройка графика работы, отпусков, перерывов; контроль перегрузки |
| Интеграция с СМС/мессенджерами | Отправка напоминаний, подтверждений, опросов满意度 |
| Отчёты по посещаемости | Анализ явки/неявки по врачам, времени, услугам |

### Функциональная модель (контекстная диаграмма)

```
[Пациент] → (записывается, подтверждает, оплачивает) → 
[МИС "Дента-Про"] ← (управляет, отчитывается) ← [Администратор]
                                 ↓
                           [Врач-стоматолог]
                                 ↓
                      (план лечения, карта, снимки)
```

## Часть 3. Роли пользователей и их проблемы

| Роль | Проблема |
|------|----------|
| **Пациент** | «Я забываю о записи к стоматологу, особенно если записался за 2 недели. Мне никто не напоминает, и я теряю время и деньги из-за штрафов за неявку». |
| **Администратор** | «Мы обзваниваем до 50 пациентов в день вручную. Это занимает 2–3 часа. При этом 30% всё равно не приходят, потому что не дозвонились или забыли». |
| **Главный врач / управляющий** | «Я не вижу реальную статистику неявок по каждому врачу и времени суток. Не могу принять решение — вводить предоплату или ужесточать запись». |

---

## Часть 4. ИТ-решения для каждой проблемы

1. **Пациент** → Автоматические напоминания за 48ч, 24ч и за 2ч до визита (СМС + push в Telegram/WhatsApp) с кнопкой подтверждения.
2. **Администратор** → Автообзвон и авто-СМС через интеграцию с сервисом голосовых роботов (например, Звонок.ру) и формирование листа подтвердивших/неподтвердивших.
3. **Главный врач** → Дашборд «Анализ неявок» с фильтрами по врачу, услуге, дню недели, часу; авторасчёт экономических потерь.

## Часть 6. Пояснительная записка (обоснование приоритетов)

**Приоритет №1 — «Умное напоминание: 3 касания до визита»**  
Прямое влияние на ключевую проблему клиники — 30% неявок. Быстрое внедрение (32 ч), высокий ROI. Без этого шага дальнейшая аналитика теряет смысл — сначала надо снизить потери, потом их измерять.

**Приоритет №2 — «Автообзвон + Smart-лист неподтвердивших»**  
Снижает ручной труд администраторов, которые перегружены. Позволяет переиспользовать освободившиеся слоты. Но требует базовой функции напоминаний (№1) как фундамента.

**Приоритет №3 — «Аналитика неявок»**  
Важна для стратегии, но не даёт мгновенного эффекта. Её ценность раскрывается после автоматизации напоминаний — тогда дашборд покажет остаточный % неявок и поможет точечно донастроить политику (предоплата, депозит, блокировка записи). Поэтому приоритет — средний.

**Итоговая рекомендация:** внедрять Feature 1 → Feature 2 параллельно (бэкенд общий), Feature 3 — в следующем спринте.

## Часть 7. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Неявка (No-show)** | Запланированный визит пациента, который не состоялся без предварительной отмены |
| **МИС** | Медицинская информационная система (учёт пациентов, расписания, карт, финансов) |
| **Автообзвон (робот-обзвон)** | Автоматическая телефонная рассылка с голосовым сценарием без участия оператора |
| **Smart-лист** | Динамический список пациентов, сформированный по правилам (например, «не подтвердили запись за 12 часов») |
| **ROI** | Return on Investment — окупаемость вложений |
| **ETL** | Extract, Transform, Load — процесс подготовки данных для аналитики |
| **MVP** | Minimum Viable Product — минимально жизнеспособная версия функции |
